# Cleanergy LPG Pump Interface & Simulation System

## Overview

This project provides a monorepo for interacting with ADCENG LPG filling scales (referred to as "pumps") via RS232 serial communication. It includes:
1.  A Python application (`sim_pc.py`) to act as a host computer, polling the scale, sending commands, parsing responses, and logging transactions to a backend API.
2.  A Python script (`sim_pump.py`) to simulate the behavior of an ADCENG LPG scale for development and testing purposes.
3.  A Node.js backend server (`server.js`) using Express and Prisma to store and retrieve transaction data from the scales.
3.  A DOTNET tester (`csharp/PayGasConsoleTester.csproj`) to test the communication protocol.

The system is designed to work with actual ADCENG scales connected to a Windows PC and also allows for a fully simulated environment on a single Windows machine using virtual serial ports.

## Features

* Communicates with ADCENG LPG scales based on the "Adceng Communications Protocol."
* Simulates ADCENG scale responses for offline development and testing.
* Sends polling requests and specific commands to scales.
* Parses responses from scales to extract transaction data.
* Logs all transactions to a central database via a REST API.
* Logs operational messages and errors from `sim_pc.py` to `transactions.log`.
* Configurable serial port settings for both Windows and Unix-like systems.

## System Architecture / Components

1.  **`sim_pc.py` (PC/Host Simulator):**
    * Connects to a specified serial port (real or virtual).
    * Polls the connected scale with commands (e.g., status request).
    * Parses the scale's ASCII-based protocol.
    * Sends transaction data to the Node.js backend API upon fill completion (status code 70).
    * Handles serial connection errors and attempts to reconnect.
    * Logs activity to `transactions.log`.

2.  **`sim_pump.py` (Pump/Scale Simulator):**
    * Listens on a specified serial port (typically one end of a virtual pair).
    * Receives commands from `sim_pc.py`.
    * Sends pre-defined or simulated responses based on the Adceng protocol.
    * Allows `sim_pc.py` to be tested without a physical scale.

3.  **`server.js` (Backend API & Database):**
    * A Node.js Express server.
    * Uses Prisma ORM to interact with a database (defaults to SQLite).
    * Provides API endpoints:
        * `POST /transactions`: To receive and store new transaction data from `sim_pc.py`.
        * `GET /transactions`: To retrieve all stored transactions.
    * Database schema is defined in `prisma/schema.prisma` and migrations are in `prisma/migrations/`.

4.  **`transactions.log`:**
    * Log file generated by `sim_pc.py` recording its operations, sent/received data, and any errors.

5.  **Adceng Communications Protocol Document:**
    * (`Adceng Communications Protocol.pdf`) Defines the RS232 serial communication specifics (9600 baud, 8N1, packet structure: `$data,data*`).

## Technology Stack

* **Python 3.x:** For `sim_pc.py` and `sim_pump.py`.
    * `pyserial`: For serial port communication.
    * `requests`: For making HTTP requests from `sim_pc.py` to the backend API.
* **Node.js:** For the backend server (`server.js`).
    * `express`: Web framework.
    * `@prisma/client`: Prisma ORM client.
* **Prisma:** ORM for database interaction.
* **SQLite:** Default database used by Prisma in this setup (can be changed in `prisma/schema.prisma`).
* **(For Windows Simulation) com0com:** Virtual null-modem emulator to create linked pairs of virtual COM ports.
* **.NET:** For the DOTNET tester (`csharp/PayGasConsoleTester.csproj`).

## Setup Instructions

1.  **Clone the Repository (if applicable) or Extract Project Files.**

2.  **Setup Backend Server (`server.js`):**
    * Navigate to the directory containing `server.js` and `package.json`.
    * Install Node.js dependencies:
        ```bash
        npm install
        ```
    * Apply Prisma database migrations (this will also create the SQLite database file, typically `prisma/dev.db`):
        ```bash
        npx prisma migrate dev --name init
        ```

3.  **Setup Python Environment:**
    * Ensure Python 3.x is installed.
    * Install required Python packages (if not already done):
        ```bash
        pip install pyserial requests
        ```

4.  **com0com Setup on Windows for Simulation (IMPORTANT):**
    * **Purpose:** To allow `sim_pc.py` and `sim_pump.py` to communicate with each other on the same Windows machine as if connected by a serial cable.
    * **Installation:** Download and install `com0com` (requires administrator rights).
    * **Create a Port Pair:** Use the `com0com` Setup utility (from the Start Menu) or `setupc.exe` (run as Administrator from the `com0com` installation directory) to create a virtual COM port pair. For example, create a pair like `COM10` <=> `COM11` (or `COM12`, `COM13`, etc., if `COM10`/`COM11` are in use or cause issues).
        * You can check existing pairs with `setupc.exe list`.
        * To create the first default pair (often `CNCA0` <=> `CNCB0`): `setupc.exe install`
        * To change names (example):
            ```cmd
            setupc.exe change CNCA0 PortName=COM10
            setupc.exe change CNCB0 PortName=COM11
            ```
            *(Adjust port names as needed if you encounter "ComDB: COMx - logged as 'in use'" errors by choosing different, unused COM port numbers).*
    * **CRITICAL NOTE - Driver Signatures & Secure Boot:**
        Modern Windows versions (1607+ / Windows 10/11) enforce strict driver signing. `com0com` installs a kernel-mode driver.
        * **Issue:** If the `com0com` driver is not correctly signed or if Secure Boot is enabled and interfering, the virtual COM ports may not be created or usable, leading to `FileNotFoundError` when scripts try to open these ports.
        * **Workaround:** If you face issues with `com0com` driver installation/operation, **disabling Secure Boot in your computer's BIOS/UEFI settings** is a known workaround.
            * **Disclaimer:** Disabling Secure Boot can have security implications. Understand the risks. This is primarily a workaround for development/testing.
            * **How:** Restart PC, enter BIOS/UEFI (common keys: F1, F2, F10, F12, Del, Esc), find "Secure Boot" option (often under "Boot" or "Security"), set it to "Disabled," save and exit.
            * After disabling, (re)install `com0com` and set up your port pair.
        * Ideally, use a version of `com0com` with drivers signed in a way your system accepts with Secure Boot enabled, if available.

## Running the System

1.  **Start the Backend Server:**
    * Navigate to the directory with `server.js`.
    * Run:
        ```bash
        node server.js
        ```
    * The server should start on `http://localhost:3000`.

2.  **Run in Simulation Mode (on Windows):**
    * Ensure `com0com` is installed and a virtual COM port pair is set up (e.g., `COM10` <=> `COM11`).
    * **Start the Pump Simulator (`sim_pump.py`):**
        * Edit `sim_pump.py` to use one end of your `com0com` virtual port pair (e.g., `ser = serial.Serial('COM11', 9600, timeout=2)`).
        * Open a new terminal, navigate to the script directory, and run:
            ```bash
            python sim_pump.py
            ```
    * **Start the PC/Host Simulator (`sim_pc.py`):**
        * This script takes the COM port number (just the number part for Windows) as a command-line argument.
        * Open another new terminal, navigate to the script directory, and run (if COM10 is the other end of your pair):
            ```bash
            python sim_pc.py 10
            ```
        * `sim_pc.py` will attempt to connect to `COM10`. You should see communication between `sim_pc.py` and `sim_pump.py`, and logs in `transactions.log`. Successful transactions will be POSTed to the Node.js server.

3.  **Run with a Real Pump/Scale (Windows):**
    * Connect the actual ADCENG scale to your PC via its RS232 port (likely using a USB-to-RS232 adapter).
    * Identify the COM port assigned by Windows in Device Manager (e.g., `COM3`).
    * Ensure the Node.js server (`server.js`) is running.
    * **Start the PC/Host Simulator (`sim_pc.py`):**
        * Run `sim_pc.py`, passing the real COM port number as an argument:
            ```bash
            python sim_pc.py 3
            ```
        * `sim_pc.py` will attempt to connect to the physical scale on `COM3`.

## Testing using the DOTNET Tester

## Configuration

* **`sim_pc.py`:**
    * **Serial Port:** Passed as a command-line argument (e.g., `python sim_pc.py 3` for `COM3` on Windows, or `python sim_pc.py 003` for `/dev/ttys003` on Unix-like systems if you adapt the script).
    * **Baud Rate:** Hardcoded to `9600` (matches Adceng protocol).
    * **API URL:** Hardcoded to `http://localhost:3000/transactions`.
    * **Log File:** Hardcoded to `transactions.log`.
* **`sim_pump.py`:**
    * **Serial Port:** Hardcoded, e.g., `serial.Serial('COM4', ...)`. **You MUST edit this** to match one end of your `com0com` pair for simulation.
    * **Baud Rate:** Hardcoded to `9600`.
* **`server.js`:**
    * **Port:** Hardcoded to `3000`.
    * **Database:** Configured via Prisma (`prisma/schema.prisma`). Defaults to a SQLite file `prisma/dev.db`.

## Troubleshooting

* Check `transactions.log` for errors from `sim_pc.py`.
* Ensure the Node.js server is running and accessible.
* Verify COM port numbers are correct and not in use by other applications.
* For simulation on Windows, ensure `com0com` drivers are correctly installed and active (see Secure Boot note above if issues persist).
* Confirm serial settings (9600, 8N1) match between scripts and the actual device protocol.
